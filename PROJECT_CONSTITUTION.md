## 🎯 قواعد المساحة (Space Rules)

### القاعدة 1: مصدر الحقيقة الوحيد
أي قرار معماري، قاعدة، أو اتفاق واجهات (Interfaces) يُكتب في هذه الوثيقة قبل التنفيذ.

### القاعدة 2: نطاق واضح لكل وكيل
كل وكيل يعمل ضمن ملفات/مجلدات محددة. **ممنوع التعديل خارج النطاق إلا بتصريح مكتوب داخل المساحة.**

### القاعدة 3: سير عمل الفروع (Branch Workflow)
- كل وكيل يعمل على Branch مستقل باسم واضح (مثال: `agent-1-core`, `agent-2-memory`)
- الدمج إلى `main` يتم فقط بعد اجتياز الاختبارات والمراجعة

### القاعدة 4: جودة قبل الدمج
ممنوع الدمج إذا:
- فشل lint/format/tests
- انخفضت التغطية عن الحد المتفق عليه (80%+)

### القاعدة 5: الحذف الآمن
أي حذف لكود/ملفات يجب أن يسبقه:
- فحص التبعيات
- التأكد أنه غير مستخدم
- حذف "نهائي" وليس تعليقاً أو ترك بقايا

### القاعدة 6: التغييرات تُحفظ مباشرة وبوضوح
- كل تعديل قابل للتتبع (commits واضحة)
- تجميع التعديلات المتعلقة في Commit واحد عند تسليم مهمة مكتملة

### القاعدة 7: واجهات ثابتة
أي تغيير في API/Schema/Events يجب:
- أن يكون متوافقاً للخلف قدر الإمكان
- أو يتضمن خطة Migration موثقة

### القاعدة 8: سجل التقدم
كل وكيل يحدّث يومياً (أو عند التسليم):
1. ماذا أنجزت؟
2. ماذا تبقى؟
3. ما المخاطر/العوائق؟

### القاعدة 9: منع التعارض
عند وجود تعارض بين مخرجات وكيلين:
- المرجع: الوثائق + اختبارات التكامل
- وليس "الاجتهاد الشخصي"

## 1. نظرة عامة على النظام

### 1.1 الهدف الرئيسي
**"وكيل مطوّر برمجيات قادر على تنفيذ كل شيء"** - نظام ذكاء اصطناعي هرمي يستقبل أي مهمة معقدة، يُحلّلها، يُخطّط لها، يُنشئ وكلاء متخصصين ديناميكياً لتنفيذها، ويُتحقق من جودتها عبر وكيل ناقد قبل التسليم.

### 1.2 الفلسفة المعمارية
النظام يتبع نمط **Plan-then-Execute** مع **Hierarchical Multi-Agent Orchestration** + **Critic-based Reflection**

```
المستخدم → المدير (التخطيط) → الوكلاء الفرعيون (التنفيذ) → الناقد (التحقق) → النتيجة
```

### 1.3 المبادئ الأساسية
1. **الاستقلالية بحدود**: تنفيذ واسع بعد موافقة على الخطة
2. **الدقة قبل السرعة**: كل خطوة تُتحقق قبل الانتقال للتالية
3. **التعلم المستمر**: كل مهمة تُثري قاعدة المعرفة
4. **الشفافية الكاملة**: كل قرار مُسجّل ومُبرَّر
5. **المرونة الذكية**: فشل وكيل لا يُفشل المهمة

### 1.4 معايير النجاح
- ✅ **المدخلات = المخرجات**: ما طُلب بالضبط هو ما يُسلَّم
- ✅ **الدقة التقنية**: الكود يعمل، الأتمتة تنجح، البيانات صحيحة
- ✅ **الإبداع ضمن القيود**: حلول مبتكرة لكن آمنة
- ✅ **قابلية التدقيق**: كل خطوة موثّقة ومُفسَّرة

## 2. العمارة الهرمية للوكلاء

### 2.1 الطبقات الثلاث

```
┌─────────────────────────────────────────────────────────────┐
│                     ORCHESTRATOR AGENT                       │
│              (المدير - Groq Llama 4 70B)                    │
│  المهام: تحليل الطلب، تقسيم المهام، بناء الخطة،            │
│         تعيين الوكلاء، المراقبة، إعادة التخطيط              │
└──────────────┬──────────────────────────────┬────────────────┘
               │                              │
       ┌───────▼──────────┐          ┌───────▼──────────┐
       │  WORKER AGENTS   │          │  CRITIC AGENT    │
       │   (الوكلاء       │          │   (الناقد)       │
       │   الفرعيون)      │          │  Gemini Pro/     │
       │  Groq/Gemini     │◄─────────┤  Claude Sonnet   │
       │  Flash           │  نقد     │                  │
       └──────────────────┘          └──────────────────┘
          │    │    │
    ┌─────┴┐ ┌─┴───┐ ┌──┴──────┐
    │ كود  │ │بحث │ │أتمتة   │  ... (حتى 7 وكلاء)
    │      │ │     │ │         │
    └──────┘ └─────┘ └─────────┘
```

### 2.2 الأدوار التفصيلية

#### أ) Orchestrator Agent (الوكيل المدير)
**الهوية:**
- **النموذج:** Groq Llama 4 70B (أو Mixtral 8x22B للمهام الأكثر تعقيداً)
- **الذاكرة:** يقرأ من `global_memory.db` + `knowledge_base.md`
- **الصلاحيات:** خطة + طلب موافقة فقط، لا تنفيذ مباشر

**المسؤوليات:**
1. **فهم الطلب**: تحليل دلالي عميق لنية المستخدم
2. **التخطيط الشامل**: بناء خطة مرحلية (جدول مع 5 أعمدة)
3. **إنشاء الوكلاء**: توليد ديناميكي لـ Worker Agents
4. **المراقبة**: تتبع حالة كل وكيل
5. **إعادة التخطيط**: عند فشل وكيل 3 مرات
6. **التلخيص**: دمج مخرجات الوكلاء

**نمط التفكير:** Plan-and-Execute

#### ب) Worker Agents (الوكلاء الفرعيون)
**الأنواع الديناميكية:**
1. **Researcher Agent**: بحث ويب، جمع معلومات
2. **Code Generator Agent**: كتابة Python/JS/PowerShell
3. **Automation Agent**: تشغيل سكربتات، أتمتة متصفح
4. **Data Agent**: معالجة CSV/JSON/Excel
5. **Communication Agent**: إرسال إيميلات، تيليغرام
6. **Visual Agent**: معالجة صور، توليد بـ AI
7. **QA Agent**: اختبار الكود

**نمط التفكير:** ReAct (Reason → Act → Observe → Repeat)

#### ج) Critic Agent (الوكيل الناقد)
**الهوية:**
- **النموذج:** Gemini 2.5 Pro أو Claude 3.7 Sonnet
- **المعايير الأربعة:**
  1. الدقة (Accuracy)
  2. الاكتمال (Completeness)
  3. الالتزام بالنمط (Compliance)
  4. خلو الأخطاء (Error-free)

**نمط التفكير:** Reflection Pattern

## 5. تدفق العمليات

### 5.1 تدفق المهمة الكاملة

```
1. استقبال الطلب (User Input)
   ↓
2. Orchestrator: فهم الطلب + RAG من الذاكرة
   ↓
3. Task Planner: بناء الخطة المرحلية
   ↓
4. عرض الخطة → انتظار الموافقة
   ↓
5. Worker Factory: إنشاء الوكلاء (1-7)
   ↓
6. Execution Engine: تنفيذ متزامن
   ↓
7. Critic Agent: تقييم المخرج
   ↓
8. تسليم النتيجة أو إعادة التنفيذ
```

### 5.2 آليات الحماية من الفشل
- **كشف الحلقة اللانهائية**: 3 أفعال متطابقة → توقف
- **إعادة المحاولة الذكية**: 3 محاولات بطرق مختلفة
- **نقاط الاستئناف**: حفظ بعد كل مرحلة ناجحة
